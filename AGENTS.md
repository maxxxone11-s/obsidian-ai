# Правила агента для Obsidian Vault

Это Obsidian Vault для обучения **Python Backend** и **AI**.

## Основные правила

### Язык и форматирование
1. **Все заметки писать на русском языке**
2. **Использовать Markdown**
3. **Использовать Obsidian-ссылки** формата `[[Название темы]]`
4. **Не создавать дубликаты** заметок
5. **Перед созданием новой заметки** проверять, нет ли уже похожей

### YAML Frontmatter для учебных тем

Для каждой учебной темы использовать следующую структуру:

```yaml
---
type: concept
area: [область знания]
status: [статус]
created: [дата создания]
updated: [дата обновления]
tags: [теги]
---
```

### Возможные статусы

- **draft** — черновик
- **learning** — тема изучается
- **needs_review** — тема понята слабо
- **learned** — тема закреплена

## Структура заметки

Каждая concept-заметка должна помогать сохранить понимание, а не только определение. Начиная с Knowledge Synchronization Protocol 2.1 каждая синхронизированная concept-заметка должна отвечать на четыре вопроса:

1. Что это?
2. Зачем это существует?
3. Какую проблему решает?
4. Какие ошибки часто делают?

Каждая concept-заметка должна содержать следующие секции в указанном порядке:

```markdown
# Название темы

## Академическое определение
Точное технически корректное определение: что это за концепция.

## Инженерное назначение
Где концепция используется в реальной системе, какую роль выполняет и зачем нужна инженерам.

## Причина существования
Почему концепцию пришлось ввести, какое ограничение она решает и что было бы без неё.

## Простое объяснение
Объяснение простым языком без академической формальности.

## Как это работает
Пошаговый внутренний механизм.

## Пример
Практический пример, желательно из AI, ML, PyTorch или Backend.

## Типичные ошибки
Частые заблуждения новичков и ошибки из предыдущих учебных сессий.

## Связанные темы
Только wikilinks, без пояснений.

## Вопросы для проверки
Вопросы для самопроверки понимания.

## Следующие темы
Концепции, которые естественно изучать дальше.
```

## Процесс добавления новой темы

После добавления темы необходимо обновить:

1. **Соответствующий Index файл** — добавить ссылку на новую тему
2. **02 Progress/Learning Log.md** — записать о добавлении темы
3. **01 Roadmap/Roadmap.md** — обновить дорожную карту обучения

## Validation Rules перед KNOWLEDGE_EXPORT

### Knowledge Areas

Knowledge Areas считаются стабильными доменами верхнего уровня.

Примеры:

- Python Backend
- Machine Learning
- Statistics
- Neural Networks
- Transformers
- PyTorch
- LangGraph
- RAG
- LLM Engineering
- AI Agents
- Algorithms & Data Structures

### Area Validation

Перед импортом любого **KNOWLEDGE_EXPORT** агент обязан:

1. Проверить, что `module.area` существует среди стабильных Knowledge Areas.
2. Никогда не создавать новую Knowledge Area автоматически.
3. Никогда не переименовывать существующую Knowledge Area автоматически.
4. Никогда не выбирать между похожими Knowledge Areas автоматически.
5. Если область неизвестна — остановить импорт и спросить пользователя.

Пример ответа:

```markdown
❌ Import stopped.

Unknown Knowledge Area:
Graph Theory

Possible actions:

- Create a new Knowledge Area
- Import into an existing Knowledge Area
- Cancel import
```

### Similar Area Detection

Если входящая область похожа на существующую, импорт нельзя продолжать автоматически.

Пример:

```yaml
Incoming:
area: Algorithms

Existing:
Algorithms & Data Structures
```

Агент обязан запросить подтверждение вместо предположения.

### Folder Creation

Папки внутри уже существующей и подтвержденной Knowledge Area можно создавать автоматически.

Пример:

```yaml
Existing area:
Algorithms & Data Structures

Incoming folder:
Algorithms/Graph Theory/DAG
```

Разрешено:

- создать `Graph Theory/`
- создать `DAG/`
- создать index-файлы
- создать concept-заметки

Это действие не требует дополнительного подтверждения пользователя.

### Automatic Import Rules

После подтверждения пользователем целевой Knowledge Area агент может:

- создавать папки
- создавать indexes
- создавать concept-заметки
- создавать backlinks
- обновлять Dashboard
- обновлять Knowledge Map
- обновлять Roadmap
- обновлять Learning Log

### Forbidden Operations

Агенту запрещено:

- создавать новые Knowledge Areas
- переименовывать существующие Knowledge Areas
- автоматически объединять Knowledge Areas
- переносить заметки между Knowledge Areas без подтверждения

### Alias Resolution

Aliases являются только подсказками.

Пример:

```yaml
Incoming:
Graph Theory

Possible destination:
Algorithms & Data Structures
```

Агент обязан спросить:

> Do you want to import this module into Algorithms & Data Structures?

Только после явного подтверждения импорт можно продолжать.

### Guiding Principle

Knowledge Areas верхнего уровня стабильны.

Внутренняя иерархия папок гибкая и может развиваться автоматически.

## Knowledge Synchronization Protocol 2.1

### Philosophy

Obsidian Vault является единственным источником истины.

**KNOWLEDGE_EXPORT не импортируется, а синхронизируется.**

Vault не должен содержать несколько версий одной и той же концепции.

Vault не является словарём. Vault является инженерной базой знаний.

Каждая concept-заметка должна помогать ответить не только:

- "Что это?"

но и:

- "Почему этот компонент существует внутри архитектуры?"
- "Какую инженерную проблему он решает?"

### Synchronization Strategy

Каждый `KNOWLEDGE_EXPORT` рассматривается как синхронизация, а не append-операция.

Начиная с версии 2.1 каждая синхронизированная concept-заметка должна стремиться к richer educational structure: определение, инженерное назначение, причина существования, простое объяснение, механизм, пример, ошибки, связи, вопросы и следующие темы.

### Step 1: Locate the Concept

Перед созданием заметки агент обязан найти существующую концепцию по:

- `topic`
- aliases
- wikilinks
- совпадающим или похожим названиям файлов

Если концепция уже существует:

- не создавать дубликат;
- обновлять только существующую заметку.

### Step 2: Merge

При синхронизации агент обязан merge-ить материал в существующую структуру заметки.

Запрещено добавлять второй блок той же концепции в конец файла.

Если concept-заметка уже существует:

- сохранять ручные заметки пользователя;
- merge-ить новую информацию в существующие секции;
- создавать отсутствующие секции из canonical-структуры 2.1;
- не дублировать секции;
- улучшать объяснения, если новый материал точнее или понятнее;
- не добавлять вторую копию той же концепции.

### Step 3: Deduplicate Sections

Для каждой синхронизированной concept-заметки допустимы только по одной копии следующих смысловых секций:

- Academic Definition
- Engineering Purpose
- Reason for Existence
- Summary
- Simple Explanation
- Why It Matters
- How It Works
- Example
- Common Mistakes
- Related
- Review Questions
- Next Topics

В русскоязычных заметках им соответствуют:

- Академическое определение
- Инженерное назначение
- Причина существования
- Кратко / Summary
- Простое объяснение
- Зачем это нужно
- Как это работает
- Пример
- Типичные ошибки
- Связанные темы
- Вопросы для проверки
- Следующие темы

Если дубликаты секций уже существуют, агент обязан объединить их автоматически без потери смысла.

### Step 3.1: Canonical Concept Structure 2.1

Каждая новая или синхронизированная concept-заметка должна содержать canonical-секции в этом порядке:

1. `## Академическое определение`
2. `## Инженерное назначение`
3. `## Причина существования`
4. `## Простое объяснение`
5. `## Как это работает`
6. `## Пример`
7. `## Типичные ошибки`
8. `## Связанные темы`
9. `## Вопросы для проверки`
10. `## Следующие темы`

Смысл секций:

- `Академическое определение` — точное технически корректное определение; отвечает "что это?".
- `Инженерное назначение` — роль концепции в реальной системе, место в архитектуре и ответственность; отвечает "зачем это инженерам?".
- `Причина существования` — мотивация появления концепции, ограничение, которое она решает, и что было бы без неё.
- `Простое объяснение` — объяснение простым языком для обучения.
- `Как это работает` — внутренний механизм по шагам.
- `Пример` — практический пример, желательно из AI, ML, PyTorch или Backend.
- `Типичные ошибки` — частые заблуждения новичков и ошибки из предыдущих учебных сессий.
- `Связанные темы` — только wikilinks, без пояснений.
- `Вопросы для проверки` — вопросы для самопроверки.
- `Следующие темы` — естественные следующие концепции.

### Step 3.2: Quality Rules

`Инженерное назначение` и `Причина существования` обязательны для новых синхронизаций.

Если этих данных нет в `KNOWLEDGE_EXPORT`, агент может вывести их только когда это очевидно из учебного материала.

Если вывод не очевиден, секцию нужно оставить пустой, а не выдумывать информацию.

### Step 4: No Import Sections

В учебные заметки запрещено вставлять секции вида:

- `Import`
- `KNOWLEDGE_EXPORT`
- `Imported from`
- `Импорт KNOWLEDGE_EXPORT`
- `История импорта`

История синхронизации хранится только в:

- `02 Progress/Learning Log.md`

### Step 5: Preserve Manual Notes

Если существующая concept-заметка содержит ручные заметки пользователя:

- сохранять их whenever possible;
- merge-ить AI-generated content в текущую структуру;
- не перезаписывать пользовательские заметки без необходимости.

### Step 6: Stop on Destructive Merge

Если синхронизация может уничтожить информацию или смысловой нюанс:

- остановиться;
- сообщить пользователю, что именно конфликтует;
- запросить решение.

### Step 7: Update Overview Files

После каждой синхронизации агент обязан обновить:

- `00 Dashboard/Dashboard.md`
- `00 Dashboard/Knowledge Map.md`
- `01 Roadmap/Roadmap.md`
- `02 Progress/Learning Log.md`
- соответствующие файлы в `Indexes/`

### Step 8: Run Validation

После синхронизации агент обязан проверить, что в заметке есть не более одной копии каждой допустимой canonical-секции.

Если дубликаты обнаружены:

- объединить их автоматически;
- оставить одну итоговую секцию.

Начиная с версии 2.1 агент также обязан проверить наличие mandatory-секций:

- `Академическое определение`
- `Инженерное назначение`
- `Причина существования`
- `Простое объяснение`
- `Как это работает`
- `Пример`
- `Типичные ошибки`
- `Связанные темы`
- `Вопросы для проверки`
- `Следующие темы`

### Principle

One concept.

One note.

One truth.

Every concept should preserve understanding, not just definitions.

## Обязательные обновления после KNOWLEDGE_EXPORT

После каждого импорта **KNOWLEDGE_EXPORT** агент обязан автоматически обновить:

1. **00 Dashboard/Dashboard.md**
2. **00 Dashboard/Knowledge Map.md**
3. **01 Roadmap/Roadmap.md**
4. **02 Progress/Learning Log.md**
5. **Соответствующий файл в Indexes/**

### Правила обновления Dashboard

- Пересчитать все учебные концепции.
- Не считать служебные файлы:
  - `AGENTS.md`
  - `README.md`
  - `Templates/*`
  - `Indexes/*`
  - `00 Dashboard/*`
  - `01 Roadmap/*`
  - `02 Progress/*`
  - `Inbox/*`
- Считать только заметки с `type: concept`.
- Посчитать:
  - всего концепций
  - `status: learned`
  - `status: learning`
  - `status: needs_review`
  - `status: draft`
- Обновить таблицу **Статистика**.
- Обновить блок **Последние обновления**.
- Добавить дату импорта.
- Указать название импортированного модуля.

### Правила обновления Roadmap

- Если `KNOWLEDGE_EXPORT` содержит `module.name`, агент обязан обновить статус этого модуля.
- Если большинство ключевых `concepts` имеют `status: learned`, модуль считается пройденным.
- Если часть `concepts` имеет `needs_review`, модуль считается требующим повторения.
- Если большинство `concepts` имеют `draft` или `learning`, модуль считается в процессе.
- Roadmap нужно обновлять не только по отдельным концепциям, но и по модулю целиком: статус модуля, общий прогресс, завершение/активность следующего этапа и темы, требующие повторения.

### Правила обновления Knowledge Map

- Добавить новые `concepts` в соответствующий раздел `area`.
- Не дублировать ссылки.
- Сохранять удобный порядок.

### Правила обновления Learning Log

- Добавить запись с датой.
- Указать импортированный `module.name`.
- Перечислить новые и обновлённые `concepts`.
- Указать `weak_spots` из `learning_state`, если они есть.
- Указать `recommended_next_step`, если он есть.

## Работа с материалом из ChatGPT

Если пользователь вставляет материал из ChatGPT, нужно:

1. **Выделить главную тему**
2. **Понять связанные темы**
3. **Определить уровень понимания**
4. **Сохранить самое понятное объяснение**
5. **Добавить типичные ошибки** пользователя
6. **Создать связи между темами** через Obsidian-ссылки

### Классификация Algorithms & Data Structures

## SYNC_PACKAGE Protocol

**SYNC_PACKAGE** — специальный протокол для обновления базы знаний структурированными пакетами данных из ChatGPT или других источников.

### Когда активировать

Если пользователь вставляет блок, начинающийся с:
```
SYNC_PACKAGE:
topic: ...
```

это означает структурированное обновление базы знаний.

### Обязательные действия агента

1. **Извлечь поля пакета:**
   - `topic` — название концепции
   - `area` — область знания
   - `status` — статус изучения
   - `confidence` — уровень уверенности (0-1)
   - `related` — связанные концепции
   - `summary` — краткое описание
   - `simple_explanation` — простое объяснение
   - `examples` — примеры кода
   - `common_mistakes` — типичные ошибки
   - `review_questions` — вопросы для проверки

2. **Создать или обновить основную заметку:**
   - Новая заметка: использовать [[Templates/Concept Template]]
   - Существующая заметка: аккуратно дополнить, не перезаписывать
   - Правильное размещение: `[area]/[topic].md`
   - YAML frontmatter обновить согласно пакету

3. **Создать stub-заметки для связанных концепций:**
   - Для каждой записи в `related`
   - Если заметка не существует
   - Со ссылками на основную концепцию

4. **Обновить Index файл:**
   - Добавить ссылку на новую концепцию
   - Обновить в соответствующем `Indexes/[area] Index.md`
   - Добавить описание или категорию

5. **Обновить учебные логи:**
   - `02 Progress/Learning Log.md` — записать о синхронизации
   - `01 Roadmap/Roadmap.md` — отметить прогресс по фазам
   - Обновить статистику заметок

6. **Правила сохранения данных:**
   - ✅ Сохранить всё новое содержимое
   - ✗ Не удалять существующий материал без разрешения
   - ⚠️ При конфликтах — предложить план действий

7. **Вывести отчет:**
   - Список всех созданных/обновленных файлов
   - Количество новых заметок
   - Новые связи между концепциями
   - Обновленную статистику

### Пример пакета

```
SYNC_PACKAGE:
topic: Gradient Descent
area: ml-basics
status: learned
confidence: 0.85
related: [Linear Regression, Optimization, Neural Networks, Learning Rate]
summary: Алгоритм оптимизации для нахождения минимума функции потерь
simple_explanation: Шаги вниз по склону функции к минимуму
examples:
  - Простой пример градиентного спуска для линейной регрессии
  - Пример с визуализацией шагов оптимизации
common_mistakes:
  - Неправильный выбор learning rate
  - Зависание в локальных минимумах
review_questions:
  - Что происходит при очень большом learning rate?
  - Как выбрать правильный learning rate?
```

### Совместимость с Knowledge Sync

SYNC_PACKAGE интегрируется с системой Knowledge Sync:
- Использует структуру из [[Templates/Sync Package.md]]
- Может быть преобразован в JSON при необходимости
- Совместим с автоматизацией в будущем
